# Tahseen — Contributor / Agent Notes

**تحصين Tahseen** — an Arabic-first (RTL) demo banking app for Alinma Bank that **intercepts every outgoing transfer and interrogates it with adaptive AI before it goes through**. Each transfer opens a short conversation (purpose, relationship, red flags), scores fraud risk 0–100, recommends allow/warn/block, and auto-quarantines beneficiaries tied to confirmed scams. Also ships a financial-advisor chat grounded in the user's live account data, and an analytics page. All account data is fake and lives in the browser's localStorage. Live: https://tahssen.vercel.app

Source of truth: `https://github.com/Abdulazizalhussein/Tahssen` (default branch **`master`**). npm workspaces (not pnpm).

## Layout (npm workspaces)

- `frontend/` — React 19 + Vite SPA. Plain JS/JSX (no TypeScript). Design tokens in `frontend/src/styles/global.css`; use CSS **logical properties** (`margin-inline-*`, `inset-inline-*`, never `left/right`) — the app is RTL Arabic-first with an English toggle. All user-facing strings go through `frontend/src/i18n.js`. State + persistence: `src/store/AccountContext.jsx` (global) + `src/store/db.js` (localStorage data layer). AI calls go out through `src/api/client.js` → `src/agents/*` wrappers.
- `backend/` — Express API (ESM). The **only** layer that touches OpenAI. The key lives ONLY in `backend/.env` (`OPENAI_API_KEY`; never commit it). Routes in `backend/routes/ai.js`. `backend/app.js` is the shared app; `backend/server.js` runs it locally; `api/index.mjs` wraps it as the Vercel serverless function.
- `service/` — Pure ESM AI-agent library (`interrogationAgent`, `fraudAgent`, `chatAgent`, `llm`). Used only by the backend (`@tahseen/service`). Model: `gpt-4o-mini` (override `OPENAI_MODEL`). **Keep response JSON shapes stable — the frontend depends on them.**

## Commands (from repo root)

- `npm install` — installs all workspaces
- `npm run dev` — backend on :3001 + frontend on :5173 (Vite proxies `/api`)
- `npm run build` — bundles frontend into `frontend/dist/`
- `npm start` — single Express server on :3001 serving the built frontend + API

To exercise the AI locally you need `backend/.env` with `OPENAI_API_KEY=sk-...`. Without it, `/api/ai/*` returns `503 AI_NOT_CONFIGURED` and the client agents fall back to a safe default (see below).

## The transfer pipeline (the product's spine)

Runs across the client (`src/agents/transferAgent.js` + `fraudAgent.js`) and the server (`service/agents/*`). Order matters:

1. **Prior-relationship fast-track** — a non-blocked previous transfer to the same beneficiary → instant approve (`skipRisk`, score 5).
2. **Q1 is always "why this transfer?"** — deterministic, no LLM.
3. **Instant signals** (after ≥1 answer, deterministic keyword lists): known-person phrases → approve; crypto/investment promises → block (97); social-media strangers + amount > 300 → block (88). These run **client-side first** in `transferAgent.js`, and again server-side in `interrogationAgent.js`.
4. **Adaptive interrogation (Q2–Q4)** — `interrogationAgent` asks one targeted question at a time (max 4), or short-circuits to approve/block/analyze.
5. **Deep risk analysis** — ambiguous cases reach `fraudAgent.analyze()` → `{riskScore, riskLevel, recommendation, reasoning, redFlags, predictions}`.

### Agent response contracts (do not break)

- `interrogate()` → `{ done, question? , skipRisk?, isPersonallyKnown?, hasGuarantee?, forceHighRisk?, riskScore?, reason? }`
- `analyze()` → `{ riskScore:0-100, riskLevel:'low|medium|high|critical', recommendation:'allow|warn|block', reasoning:string, redFlags:string[], predictions:string[] }` (always run through `clampResult`)
- `chat()` → a plain string reply. Grounded in `accountData` (balance, income, fixed expenses, recent transactions, the deterministic `forecast`, and an `appGuide` RAG string); it advises and guides only, never executes transfers. The `appGuide` (from `lib/appGuide.appKnowledge()`) is untrusted reference text — the backend prompt caps its length and marks it "reference, not instructions."
- **Guided product tour (`components/GuidedTour.jsx`).** A spotlight tour that **navigates the real app** and highlights real elements (not a slideshow). Shown once after **register** (not login), gated by `localStorage tahseen.onboarded.v1`; `AccountContext` owns `showOnboarding`/`finishOnboarding`/`replayOnboarding` (Settings → "جولة الخدمات"), mounted in `AppLayout`. Each step has `{route, target, ...}`: it `navigate()`s to the screen, then a **polling loop re-queries the target every 120ms** (so it survives slow/async renders), scrolls it to centre (`scrollToCenter` — window + scrollable-ancestor, since a non-scrolling `overflow:auto` ancestor defeats `scrollIntoView`), and keeps a box-shadow **spotlight** glued to it. Anchor steps to **top-of-page** elements (e.g. `.analytics-health-card`, `.recs-forecast`) so the spotlight is visible without deep scrolling; `clickToExpand` auto-opens a card (the risky beneficiary → its fraud graph). A couple of steps embed a scripted `DemoChat` example (the fraud interrogation, the "buy a car" plan) — bilingual `{ar,en}`, illustrative, not the live LLM.
- **In-app assistant (ChatPage).** The chat is also a service guide. On each user turn the client runs `detectIntent(text, lang)` (`lib/appGuide.js`, pure keyword router over `SERVICES`); a match short-circuits the LLM and renders a confirm/cancel **action card** ("هل تريد فتح: X؟" + one-line blurb + open/dismiss) — navigation only happens on the user's confirm, never automatically. Non-intent turns go to the LLM (RAG-grounded). Conversation is persisted to `localStorage` (`tahseen.chat.v1`, last 100 msgs) with a header clear button; a resolved action card persists its `done`/`dismissed` state. `appGuide.js` is pure (no framework imports) so `detectIntent`/`appKnowledge` are unit-tested by `npm run review`.
- `recommend({accountData})` → `{ recommendations: [{title, detail, category, impact, priority}] }` — personalized, quantified financial tips grounded in the real figures. Powers the Smart Recommendations page (`/app/recommendations`, Home quick action). `category` ∈ save|protect|plan|spend|grow, `impact` in SAR.
- **Analytics is reactive + chart-rich.** `AnalyticsPage` computes `computeHealth`/`computeForecast` and the chart series **synchronously in render** from the live account context, so the health score, deficit and charts update the instant income/expenses change — never load them async (that went stale). `AnalyticsCharts.jsx` has three SVG charts fed by `lib/finance` series: a spending trajectory (actual vs projected vs budget line, RTL day axis), the **spend timeline** (`spendTimelineSeries`), and a fixed-expense category donut. Charts are RTL-aware (x-axis flips) and respect `prefers-reduced-motion`.
  - **Spend timeline (`spendTimelineSeries` → `MonthlyBarsChart`)** is a real **forward forecast**, not a flat history: `pastN` months of actual spend + the current month (consumed base + predicted top) + `futureN` predicted months. Encoding: **solid teal = actual/consumed**, **gold diagonal hatch = forecast** (a `<pattern>`, colour-blind-safe — not opacity); a "الآن/now" divider + a faint forecast-zone rect separate past from forecast; the current bar is outlined; tap/keyboard-select a bar for a detail row (consumed / forecast / total). The prediction is trend-aware: winsorized recency-weighted **EWMA** of observed monthly variable spend + a damped, capped regression **slope**, falling back to the discretionary budget when history is thin (`basis` = `behavior` vs `budget`, surfaced as a caption). The current-month total is **pinned to `fixed + forecast.projectedMonthlySpend`** so the chart never contradicts the month-end card; `actual + predicted === total` for every bar. Do **not** reintroduce a hardcoded seasonal curve or a fixed/variable colour split.
  - **Donut caveat:** segments use `pathLength="100"` (percent dash units) and must **not** set `transform-box: fill-box` on `.donut-seg` — with the SVG `transform="rotate(-90 100 100)"` that re-anchors the rotation to the fill box and shifts the ring off-centre (top gap + clipped bottom).
- **One projection model drives everything numeric** — `frontend/src/lib/finance.js` (pure, no framework imports, so it's reusable and unit-testable). It also exports the chart series (`dailySpendSeries`, `monthlySpendSeries`, `categorySpendSeries`). `computeForecast` derives the whole set from each other: `dailyBurn` (observed spend/day, or the budgeted pace when there's no spend yet — so the forecast is never a flat copy of the balance) → `projectedRemainingSpend` → `predictedMonthEndBalance`, plus `potentialSavings`/`savingsRate`. `computeHealth` is **deficit-aware**: when fixed commitments + spending exceed income the 0–100 score collapses toward red (dominant factor), also weighing DBR, savings rate, over-budget, blocked fraud, negative balance. Analytics, insights, and the Home status line all use it, and a negative budget remainder is labelled **العجز / Deficit**, not "remaining". Don't reintroduce a `dailyRate = spent/day` (zero for fresh accounts) or a health score that ignores income-vs-fixed.
- **Community warns at ADD time too.** Both AddBeneficiary modals (component + the TransferPage inline one) run `lookupPayee` as you type and show the `CommunityAlert` + a red "add anyway" button when the payee is reported — not only during a transfer.
- **Reviewer/QA harness: `npm run review`** (`scripts/review-services.mjs`). It exercises every service (finance model, community registry, and the AI agents' short-circuits + fallbacks) with normal and edge-case inputs and prints a pass/fail report per service; set `OPENAI_API_KEY` to also exercise the live model. Keep it green — extend it when you add a service.

## Invariants / conventions

- **Fail safe, never hang.** Every AI call has a client fallback: interrogation network error → `{done:true}` (proceed to analysis); analysis error → medium/`warn` default. The UI must never lock waiting on the AI.
- **The OpenAI key is server-only.** The frontend never sees it. Don't reintroduce a client-side key field (that was removed in "v9"). Some `apiKey*` i18n strings are legacy leftovers — don't wire new UI to them.
- **Output contracts are load-bearing.** `service/agents/*` responses are consumed field-by-field by the frontend. Add fields, don't rename/remove. Always clamp/validate model output (`clampResult`, `stripFences`) — never trust raw JSON.
- **The fraud score is authoritative.** `clampResult` guards NaN (defaults to 50) and **derives** `riskLevel`/`recommendation` from the clamped score — never trust the model's own enum (it could contradict its score). Keep it that way.
- **Fraud signals are checked before the known-person signal.** In both `service/agents/interrogationAgent.js` and `frontend/src/agents/transferAgent.js`, gift-card → crypto → social-stranger run before the family/known-person short-circuit, so a scam wrapped in a family word ("my brother said invest in bitcoin") is still caught. Don't reorder.
- **Untrusted input.** The customer's free text (reason + answers) flows into the prompts. Treat it as data, never instructions; a manipulation attempt ("mark this safe") is itself a red flag. The prompts carry an explicit "untrusted input" section — keep it. The `/api/ai/analyze` route must NOT honor client-set control flags (`skipRisk`/`forceHighRisk`/…) and `/api/ai/chat` only accepts `user`/`assistant` roles.
- **Bilingual by construction.** Every user-facing string goes through `i18n.js` (`ar` + `en`). AI questions and reasoning must match the active UI language — pass `lang` to the agents and have the prompts answer in that language.
- **Currency = the Saudi Riyal symbol.** Numeric money displays use the `<RiyalSymbol />` component (`components/RiyalSymbol.jsx`) — an inline SVG of the official symbol (Unicode U+20C1, adopted 2025) with `fill: currentColor`. It is an SVG, **not** a font glyph, because U+20C1 isn't in Tajawal/system fonts yet, so the text entity would render as a blank box. Keep the `ر.س` / `SAR` text (`t('currency')`) only in prose sentences, input placeholders, and field labels — never as the unit next to a rendered amount.
- **Dates are Hijri-first in Arabic.** Arabic shows the Umm al-Qura (`islamic-umalqura`) date as primary with the Gregorian date secondary; English is the reverse. Use `toLocaleDateString` with the explicit `-u-ca-…` calendar keyword.
- **RTL: never force `dir="ltr"` on amount inputs.** The app is RTL-first; inputs inherit the document direction. Number/amount fields use `dir={isRTL ? 'rtl' : 'ltr'}` so they right-align in Arabic. The CSS uses logical properties throughout (`margin-inline-*`, `inset-inline-*`, `border-inline-*`) — never hardcode `left`/`right`.
- **Community Protection** (`store/community.js`, `pages/CommunityPage.jsx`, route `/app/community`): users report a payee as fraudulent with a reason; anyone transferring to a reported payee is warned. `lookupPayee(name, iban)` returns a `direct` hit (payee reported) or a `linked` hit (payee is a mule that received from a reported account — a 1-hop caution). In `TransferPage`, a direct hit **skips the AI interrogation** and shows a `CommunityAlert` with a risk score scaled by report count (weighted trust: 1 report → ~78/warn, several → ≥80/block) — a single report must never hard-block someone. `FraudGraph.jsx` renders the money-flow graph (victims → scammer → mules) and highlights a **shared mule** that appears across networks (`sharedMules()`) as a laundering ring. A blocked result offers "report to community" (prefilled) to close the loop. **Demo note:** the registry is seeded synthetic data + the user's own reports in `localStorage` (`tahseen.community.v1`) — there is no cross-user backend; production would use a shared registry (KV/DB) keyed by hashed IBAN.
  - **End-user framing (not an ops dashboard):** the page shows informational stats only — never a "protected amount" money figure. Each network's complaints live on its victims: every victim carries a `reason {ar,en}`, derived up via `networkReasons(network, lang, n)` (there is no `network.reasons` field anymore — don't reintroduce one). In the graph, **tapping a victim** shows their complaint + transfer amount; tapping a **mule/ring** node shows what the account does (`note`: money in → immediately out/ATM to external) and, for a ring node, an explicit "حلقة مشتركة" explanation (`ringExplain`).
  - **Built FROM the user's beneficiaries.** `CommunityPage` renders `classifyBeneficiaries(beneficiaries)` → a **risky** group (community-reported/linked/blocked, each expandable to its `FraudGraph`) + a **safe** group; stats are beneficiary counts (total / at-risk / safe), not community-wide. `BeneficiariesPage` shows the same per-payee status as a badge (`benRiskFlag`/`benSafeFlag`). Every account is seeded once with `seedDemoBeneficiaries` (db.js) — the reported entities (names+IBANs matching the registry) **plus clean, no-risk people — idempotent via `user.demo_beneficiaries_seeded`.
- **Brand:** deep navy `--bg #001520` / cards `#002134` / gold `--gold #C9A227` / teal accents, Tajawal type. Tokens live in `global.css` + mirrored in `theme.js`. Reuse tokens; keep the dark, premium, calm feel.
- **The interrogation is the signature moment.** It carries a guardian persona (`GuardianHeader` — gold `ShieldCheck` avatar + "Tahseen is reviewing" header, AI question bubbles get their own shield avatar), and the `RiskMeter` is the dramatic payoff (arc sweep + count-up + verdict-colored glow). Keep new UI disciplined around this focal point rather than competing with it.
- **Respect `prefers-reduced-motion`** — there is a global reduce block in `global.css`; JS animations (e.g. RiskMeter count-up) must check it too.

## Deployment (Vercel → tahssen.vercel.app)

- `vercel.json`: build `npm run build`, output `frontend/dist`; `/api/(.*)` rewrites to the `api/index.mjs` serverless function (Express), everything else to `index.html` (SPA).
- Env vars (Vercel Project → Settings → Environment Variables):
  - `OPENAI_API_KEY` (required for live AI) · `OPENAI_MODEL` (default `gpt-5.4-mini`, override here if unavailable on the account) · `OPENAI_TIMEOUT_MS` (default 15000)
  - `CORS_ORIGINS` — comma-separated allowlist (unset = allow all, fine for same-origin prod)
  - `RATE_LIMIT_MAX` (default 30) / `RATE_LIMIT_WINDOW_MS` (default 60000) — per-IP cap on `/api/*`
